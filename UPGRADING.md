# Upgrading openclew

openclew evolves. When the format changes, your existing docs still work — parsers
are backward-compatible. But new features expect the current format.

`openclew migrate` bridges the gap.

---

## Vocabulary changes — oc_0.9.0

Phase 2 of the `refdoc` → `ref` rename: internal code now matches the user-facing vocabulary. `lib/**` identifiers, the Rust port, tests, skills, prompts, and onboarding templates all use `ref`. The MCP `list_docs` tool enum is now `{kind: "ref"}` (was `"refdoc"`) — **breaking** for MCP clients that pass `kind` explicitly.

`lib/migrate.js` intentionally still recognizes `refdoc` — that's how it migrates legacy docs forward.

**Nothing to do on existing projects.** Parsers still accept legacy forms. The breaking change only affects MCP clients that hardcode `kind: "refdoc"` — update to `"ref"`.

---

## Vocabulary changes — oc_0.8.0

`refdoc` → `ref`. The long form is retired from user-facing prose, templates, commands, and documentation. The short form `ref` was already the CLI verb (`openclew add ref`), this release aligns the rest of the vocabulary.

**Nothing to do on existing projects.** Parsers and `migrate` still recognize `refdoc` everywhere — in front matter, prose, related_docs, and legacy file paths. Your existing docs keep working as-is.

If you want to align your own prose with the new vocabulary, a future `openclew migrate --prose` flag will offer opt-in rewriting. Until then, the rename is purely cosmetic for downstream projects.

**What moved** :
- `templates/refdoc.md` → `templates/ref.md` (the shipped reference template)
- All user-visible strings in `README.md`, `AGENTS.md`, `CLAUDE.md`, `commands/oc-*.md`, `UPGRADING.md`

**What stayed** (on purpose, removed in oc_0.9.0) :
- Internal identifiers in `lib/**` (function names, variables)
- Rust port mirror (`rust/src/template/refdoc.rs`, identifiers)
- Tests, examples, skills, prompts

## Quick version

```bash
npm install -g openclew@latest   # or: npx openclew@latest
openclew status                  # shows legacy doc count
openclew migrate                 # dry-run: what would change
openclew migrate --write         # apply
git diff                         # review
git add doc/ && git commit -m "chore: migrate docs to openclew format"
```

---

## When to upgrade

After updating openclew, run `openclew status`. If it reports legacy docs,
run `openclew migrate` to see what needs changing.

```bash
openclew migrate
# → 12 to migrate, 45 already current, 0 errors (57 total)
```

No output = nothing to do.

## How it works

`migrate` converts docs from older formats to the current openclew format.
It is **safe by default**:

- **Dry-run first** — shows what would change without touching files
- **`--write` to apply** — only modifies files when you explicitly ask
- **Git-friendly** — files are tracked, `git diff` shows exactly what changed
- **Idempotent** — running it twice produces the same result
- **Skips current docs** — only touches files that need updating

### What it converts

| Before | After |
|--------|-------|
| `R.AlphA.Doc@7.0.0` (line 1) | `clw_ref@0.6.0 · created: ... · type: ... · ...` (ref) |
| `R.AlphA.Doc@7.0.0` (line 1, log) | `clw_log@0.6.0 · date: ... · type: ... · ...` (log) |
| `openclew@0.3.0` (line 1) | `clw_ref@0.6.0` or `clw_log@0.6.0` (prefix swap) |
| `subject: Title` (plain L1) | `- **subject:** Title` (bold Markdown list) |
| `summary: ...` | `- **doc_brief:** ...` |
| `# 📋 L1 · Métadonnées` | _(removed — metadata is on line 1)_ |
| `# 📝 L2 · Résumé` / `# L2 - Summary` | `# Summary` |
| `# 🔧 L3 · Détails` / `# L3 - Details` | `# Details` |
| YAML frontmatter (`---`) | Replaced by line 1 + L1 block |
| `<!-- L1_START -->...<!-- L1_END -->` | Pure Markdown list items |
| `<div class="oc-l1">...</div>` | Pure Markdown list items |
| French statuses (see table below) | English statuses |
| French types (see table below) | English types |

### Status mapping (French → English)

| French | English |
|--------|---------|
| `Vivant` / `Actif` | `Active` |
| `Stable` | `Stable` |
| `Archivé` | `Archived` |
| `En cours` | `In progress` |
| `Terminé` | `Done` |
| `Abandonné` | `Abandoned` |

### Type mapping (French → English)

| French | English |
|--------|---------|
| `Référence` | `Reference` |
| `Analyse` | `Analysis` |
| `Fonctionnalité` | `Feature` |
| `Documentation` / `Doc` | `Doc` |
| `Déploiement` | `Deploy` |
| `Refactoring` | `Refactor` |

### What it does NOT change

- **L2/L3 body content** — only headers are normalized, your content is untouched
- **Sub-headers** — `## Objective`, `## Key points` etc. are preserved as-is
- **Files already in current format** (`clw_ref@` / `clw_log@`) — skipped entirely
- **`_INDEX.md`** — optional cache, regenerated by `openclew index` if present

## Repointing references after file moves

If you move a file (e.g. `doc/_AUTH.md` → `doc/ref/AUTH.md`), references in
other docs' `related_docs` break. Use `--repoint` to update them:

```bash
openclew migrate --repoint doc/_AUTH.md doc/ref/AUTH.md          # dry-run
openclew migrate --repoint doc/_AUTH.md doc/ref/AUTH.md --write   # apply
```

## Step by step

### 1. Update openclew

```bash
npm install -g openclew@latest
```

### 2. Re-run init (recommended)

```bash
openclew init
```

This updates the openclew block in your instruction file (AGENTS.md, CLAUDE.md,
etc.) with the latest rules. Skipping this step means your agents may still
follow outdated instructions.

### 3. Check your docs

```bash
openclew status
```

Look for the "Legacy format" line. If it says 0, you're done.

### 4. Preview changes

```bash
openclew migrate
```

Lists every file that would be converted. No files are modified.

### 5. Apply

```bash
openclew migrate --write
```

Each converted file is printed with `✓`.

### 6. Review and commit

```bash
git diff                    # inspect the changes
openclew index              # regenerate the index (optional)
git add doc/ && git commit -m "chore: migrate docs to openclew format"
```

### 7. Verify

```bash
openclew status             # should show 0 legacy docs
openclew migrate            # should show "0 to migrate"
```

---

## After migrating

- **New docs** you create with `openclew add ref` / `openclew add log` already
  use the current format. No action needed.
- **Docs created by AI agents** will follow the format they see in your codebase.
  Once your existing docs are migrated, agents will generate in the new format.
- **Empty `doc_brief`** — some old docs may have no brief after migration.
  Run `openclew status` to find them and fill them in.

## Version-specific notes

### → 0.7.0 (clw_ref@/clw_log@ prefix, pure Markdown L1, migrate command)

**Breaking changes:**

1. **Prefix change**: `openclew@` → `clw_ref@` (refs) / `clw_log@` (logs).
   The categorical prefix encodes document type directly. Old `openclew@` docs
   still parse correctly but `openclew migrate --write` converts them.

2. **Pure Markdown L1**: L1 blocks are now plain Markdown list items
   (`- **subject:** ...`) instead of HTML div wrappers or comment markers.
   Both legacy formats still parse correctly.

3. **L2/L3 headers simplified**: `# Summary` and `# Details` (no `L2 -` / `L3 -` prefix).

4. **`_INDEX.md` is optional**: Agents use `openclew peek` for dynamic doc
   discovery. The index is a human-friendly cache generated on demand via
   `openclew index`. **Re-run `openclew init`** to update the injected block
   in your instruction file — old blocks still tell agents to read `_INDEX.md`.

5. **`migrate` command included**: Previously excluded from npm. Now ships with
   the package. Run `openclew migrate` after upgrading.

6. **Naming convention**: Refs move from `doc/_SUBJECT.md` to
   `doc/ref/SUBJECT.md`. This is a convention change, not enforced — migrate
   at your own pace. Use `--repoint` to update cross-references.

### → 0.5.3 (fix: user prompts install outside projects)

Fixed: `init` outside a project directory now installs Copilot user-level
prompts correctly. Previously, the project guard blocked all prompt installation
when no `.git` or `package.json` was present.

### → 0.5.2 (user-level Copilot prompts)

`init` now installs Copilot prompt files at **two levels**:
- Project: `.github/prompts/` (workspace-scoped)
- User: VS Code profile `prompts/` folder (available in ALL projects)

User prompt paths:
- macOS: `~/Library/Application Support/Code/User/prompts/`
- Windows: `%APPDATA%\Code\User\prompts\`
- Linux: `~/.config/Code/User/prompts/`

### → 0.5.1 (Copilot prompts + bloc update + --version)

`init` now installs Copilot prompt files in `.github/prompts/`.

Re-running `init` now **updates** the openclew block in your instruction file
instead of skipping it. New rules are applied automatically on upgrade.

### → 0.5.0 (init guard + global config)

`init` now creates `~/.openclew/config.json` on every run. If you are not inside
a project directory (no `.git`, `package.json`, etc.), init stops after creating
the global config.

New command: `openclew peek` — displays the doc index.

### → 0.4.0 (format migration)

First migration release. Converts from the legacy format (YAML frontmatter,
plain `key: value` L1, emoji headers) to the openclew format (condensed line 1,
bold L1 fields, clean headers).

### → 0.3.0 (zero Python, npm publish)

Index generator rewritten in JS. Python no longer required.
If you had a `doc/generate-index.py` from an older version, `init` removes it.

## Node.js vs Rust binary

openclew is available in two forms:

| | Node.js (npm) | Rust (binary) |
|--|---------------|---------------|
| Install | `npm install -g openclew` | `brew install openclew` (planned) |
| Requires | Node 16+ | Nothing |
| Commands | All (init, add, search, peek, checkout, status, migrate, index, mcp) | Core (init, index, mcp) |
| MCP tools | search_docs, read_doc, list_docs | + create_log, create_ref |

**Which one to use:**
- **Node.js** — full CLI, all commands, current recommended install
- **Rust binary** — zero dependencies, ideal for MCP server and environments without Node.js

Both read and write the same document format. Docs created by one are
fully compatible with the other.

If both are installed, the one first in your `$PATH` wins. Run
`openclew --version` to check which is active.

## Limitations

### Parsers are backward-compatible

Even without migrating, your docs are still readable by openclew tools.
Migration improves consistency and enables new features, but is not blocking.

### Pre-commit hook from old versions

Versions before 0.3.0 installed a Python hook (`doc/generate-index.py`).
Running `openclew init` removes it automatically. If you upgraded the
package without re-running init, delete it manually and use
`openclew index` instead.
